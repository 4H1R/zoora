## Threat model context

Zoora is a multi-tenant SaaS. The auth middleware (`internal/auth/middleware.go`)
proves the caller belongs to the *host* organization and holds a *permission*, but it
does **not** prove that the specific object addressed by a URL `:id` belongs to the
caller's org. That per-object tenant/ownership check is the **service layer's** job.
The dominant finding class below is service methods that fetch by `:id` and trust it —
skipping the org-match guard that their sibling methods already apply. The canonical
"correct" guard already exists in this codebase; every plan points at it.

## Priority rationale

P1 = confirmed HIGH-confidence cross-tenant data breach or privilege escalation
reachable by an ordinary org role. P2 = confirmed but needs a privileged role, a victim
interaction, or an insecure-default deployment. P3 = same-tenant hardening or requires a
product decision.

## Findings considered and rejected (do not re-audit)

- Zarinpal payment callback forgery / amount tampering / replay — `internal/billing/service.go:263-308` re-verifies server-side against the invoice amount; idempotent. Safe by design.
- chromedp PDF rendering SSRF / template injection — renders server-built `data:` URLs via auto-escaping `html/template`. Safe.
- LiveKit webhook authenticity — signature-verified via `webhook.ReceiveWebhookEvent`. Safe.
- WebSocket auth / origin — `chathub` upgrade is JWT + revocation + origin-allow-list checked; `join` enforces `IsMember`. Safe.
- `conversations` / `livesessions` / `qa` access control — membership-gated, org-scoped. These are the *reference pattern*, not findings.
- Quiz answer-key leakage, deadline bypass, post-submit grade tampering, share-code brute force, Excel zip-bomb/formula-injection — all already defended (`quizzes`, `questionbanks`, `imports`). Safe.
- JWT `alg` confusion — `internal/auth/jwt.go:51-54` pins HMAC. Safe.
- Bot proxy SSRF — proxy URL from trusted config, not user input. Safe.
